# Architecture вҖ” Source of Truth, Derived Files, and Build Pipeline

Quick reference for what to edit, what not to edit, and how the pipeline connects them.

---

## Source of truth

These are the files you hand-author. Everything else is derived from them.

### Structured data (YAML truth)

| File | Governs |
|------|---------|
| `registry/entries.yaml` | All derived compound entries (W-numbers, ~219 entries) |
| `registry/primitives.yaml` | Primitive root inventory (~34 roots) |
| `registry/colloquial.yaml` | Colloquial contracted forms (CLQ entries) |

### Prose-heavy content (Markdown truth)

| File(s) | Governs |
|---------|---------|
| `corpus/sentences/v4-current/s*.md` | Active sentence corpus (v4 format) |
| `corpus/sentences/v1-legacy/*.md` | Legacy sentences (frozen, not re-extracted) |
| `corpus/sentences/v2-labeled/*.md` | Legacy sentences (frozen) |
| `corpus/sentences/v3-transition/*.md` | Legacy sentences (frozen) |
| `corpus/conversations/c*.md` | All conversations (C001вҖ“C008+) |
| `corpus/translations/**/*.md` | Translation stress tests |
| `spec/*.md` | Language specifications (grammar, phonology, morphology, etc.) |
| `notes/*.md` | Design notes, open questions, semantic map |
| `registry/domains.md` | Domain registry |
| `registry/roots.md` | Named roots |
| `registry/kinds.md` | Kind-term taxonomy |

### Site configuration

| File | Governs |
|------|---------|
| `mkdocs.yml` (root) | Site navigation and theme config вҖ” the only mkdocs config |
| `www/docs/**/*.md` | Hand-authored website pages (about, cheatsheet, etc.) |

### Why two formats?

YAML is truth when content is structured records with many short fields (like a spreadsheet row). Markdown is truth when content is prose-heavy narrative with long notes, analysis, and headings. The pipeline bridges the gap by extracting structured fields out of markdown into YAML for downstream processing.

## Derived files (do not hand-edit)

These are regenerated by the pipeline. Hand-edits will be overwritten.

| File(s) | Generated by | From |
|---------|-------------|------|
| `corpus/sentences.yaml` | `extract_sentences.py` | `v4-current/s*.md` + frozen v1/v2/v3 entries |
| `corpus/batches.yaml` | `extract_batches.py` | `sentences.yaml` + markdown headings |
| `corpus/conversations.yaml` | `extract_conversations.py` | `corpus/conversations/c*.md` |
| `registry/derived/w*.md` | `generate_registry_md.py` | `entries.yaml` |
| `registry/derived/index.md` | `generate_registry_md.py` | `entries.yaml` |
| `registry/colloquial.md` | `generate_colloquial_md.py` | `colloquial.yaml` |
| `www/docs/tonesu/registry/**` | `build_registry.py` | All YAML + translations |
| `site/**` | `mkdocs build` | `www/docs/` + `mkdocs.yml` |

## Build pipeline

Run from repo root:

```
python scripts/build.py
```

### Steps

| Step | Script | What it does |
|------|--------|-------------|
| 0 | `extract_sentences.py` | Extracts sentence records from v4-current markdown вҶ’ `sentences.yaml` |
| 0 | `extract_batches.py` | Derives `batches.yaml` from `sentences.yaml` + markdown headings |
| 0 | `extract_conversations.py` | Extracts conversation records from all `c*.md` вҶ’ `conversations.yaml` |
| 1 | `generate_registry_md.py` | Generates `registry/derived/*.md` + `index.md` from `entries.yaml` |
| 1 | `generate_colloquial_md.py` | Generates `registry/colloquial.md` from `colloquial.yaml` |
| 2 | `annotate_words_attested.py` | Populates `words_attested` on sentences and conversation turns |
| 3 | `derive_first_attests.py` | Derives first-attestation data; backfills `first_use` in `entries.yaml` |
| 4 | `build_registry.py` | Generates all `www/docs/tonesu/registry/` and corpus pages for the site |
| 5 | `mkdocs build` | Builds the production site into `site/` |

### Flags

- `--dry-run` вҖ” Steps 2 and 3 won't write files (safe preview)
- `--skip N` вҖ” Skip step N (e.g. `--skip 5` to skip the mkdocs build)

### Deployment

On every push to `main`, GitHub Actions (`.github/workflows/deploy.yml`):

1. Runs `python scripts/build.py --skip 5` (steps 0вҖ“4)
2. Runs `mkdocs gh-deploy --force` (builds and deploys to GitHub Pages)

No manual deployment needed. Commit + push to `main` = site updates automatically.

To preview locally before pushing: run `python scripts/build.py` and open `site/index.html`.
